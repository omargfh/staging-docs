Qu es Espacio-Conductual?

Espacio-Conductual es una herramienta de software integrada con NetLogo que permite realizar experimentos con modelos.

Espacio-Conductual ejecuta un modelo varias veces, variando sistemticamente la configuracin del modelo y registrando los resultados de cada ejecucin. A este proceso a veces se le llama exploracin de parmetros. Permite explorar el espacio de posibles comportamientos del modelo y determinar qu combinaciones de configuracin provocan los comportamientos de inters.

Si la computadora tiene mltiples ncleos de procesador, puede especificarse cuntas ejecuciones del modelo ocurrirn en paralelo.

Por qu Espacio-Conductual?

La necesidad de este tipo de experimento se hace evidente a travs de las siguientes observaciones. A menudo, los modelos tienen muchas configuraciones, cada una de las cuales puede tomar un rango de valores. Juntas, forman lo que en matemticas se llama un espacio de parmetros para el modelo, cuyas dimensiones son el nmero de configuraciones, y en el cual cada punto es una combinacin particular de valores. Ejecutar un modelo con diferentes configuraciones (y a veces incluso las mismas) puede llevar a un comportamiento drsticamente diferente en el sistema que se est modelizando. De modo que, cmo saber qu configuracin particular de valores, o qu tipos de configuraciones, producirn el tipo de comportamiento que nos interesa? Esto equivale a la pregunta: dnde - en su enorme espacio de parmetros multidimensional- funciona mejor nuestro modelo?

Por ejemplo, supongamos que desea lograrse una sincronizacin rpida entre los agentes en el modelo Fireflies. El modelo tiene cuatro deslizadores: number, cycle-length, flash-length and flashes-to-reset, con aproximadamente 2000, 100, 10 y 3 valores posibles, respectivamente. Esto significa que, hay 2000 _ 100 _ 10 * 3 = 6,000,000 posibles combinaciones de valores de los deslizadores! Probar combinaciones una por una, no es una forma remotamente eficiente de descubrir cul provocar la sincronizacin ms rpida.

Espacio-Conductual ofrece una forma mucho mejor de resolver este problema al muestrear el espacio de parmetros del modelo, no de manera exhaustiva, pero lo suficiente como para observar la formacin de relaciones entre los diferentes valores de los deslizadores y el comportamiento del sistema. Una forma de hacerlo es especificar un subconjunto de valores dentro de los rangos de cada deslizador. Consulta Sintaxis combinatoria. Espacio-Conductual ejecutar el modelo con cada posible combinacin de esos valores y registrar los resultados durante cada ejecucin. Desde NetLogo 6.4, es posible especificar conjuntos no combinatorios de valores de los deslizadores. Consulta Sintaxis de subexperimento. Despus de que todas las ejecuciones finalizan, se genera un conjunto de datos que se puede abrir en una herramienta diferente, como una hoja de clculo, una base de datos o una aplicacin de visualizacin cientfica, y explorarlos.

Al permitir explorar todo el espacio de comportamientos que un modelo puede exhibir, Espacio-Conductual puede ser un poderoso asistente para el modelador.

Comandos pre-experimento

Estos comandos se ejecutarn una vez, antes de que inicie el experimento.

Comandos go: Estos comandos se ejecutarn una y otra vez para avanzar el modelo al siguiente paso. Por lo general, esto ser el nombre de un procedimiento, como go, pero se pueden incluir los comandos que se deseen.

Si se desea que todas las ejecuciones tengan una duracin fija, simplemente se deja en blanco.

La ejecucin tambin puede detenerse debido a que los comandos go utilizan el comando stop de la misma manera que stop se usa para detener un botn Continuo. El comando stop puede usarse directamente en los comandos go o en un procedimiento llamado directamente por los comandos go. (La intencin es que el mismo procedimiento go funcione tanto en un botn como en un experimento de Espacio-Conductual). Ten en cuenta que el paso en el que se utiliza stop se considera como si hubiera sido interrumpido, por lo que no se registrarn resultados para ese paso. Por lo tanto, la prueba de detencin debera estar al principio de los comandos go, o del procedimiento, no al final.

Comandos post-ejecucin: Estos son cualquier comando adicional que se quiera ejecutar cuando finaliza la ejecucin. Generalmente se deja en blanco, pero pudiera usarse para llamar al comando export-world o para registrar los resultados de la ejecucin de alguna otra manera.

Comandos post-experimento

Estos son cualquier comando adicional que se quiera ejecutar cuando finaliza el experimento, una vez que todas las ejecuciones han finalizado.

Lmite de tiempo: Esto permite establecer una longitud fija mxima para cada ejecucin. Cuando no se desea establecer un mximo, sino que se prefiere que la longitud de las ejecuciones est controlada por la condicin de detencin, se ingresa 0.

Si se utilizan primitivas como file-open o la extensin CSV en los comandos previos o posteriores al experimento, su estado interno no se transferir a las ejecuciones y es probable que ocurran errores si se utilizan ah.

Primera: si hay mltiples ejecuciones activas, solo una de ellas estar en el primer plano y generar la actualizacin de la vista y los grficos. Las dems ejecuciones ocurrirn en segundo plano de manera invisible.

Segunda: las ejecuciones en segundo plano invisibles no pueden utilizar primitivas que solo funcionan en la interfaz grfica (GUI). Por ejemplo, una ejecucin en segundo plano no puede crear una pelcula.

Tercera: dado que las ejecuciones en paralelo avanzan de forma independiente, el formato Resultados tabulados puede contener datos entrelazados y fuera de orden. Si el modelo utiliza una gran cantidad de memoria, es posible que reducir el nmero de ejecuciones permita que estas se completen en menos tiempo, ya que el sistema puede mantener disponible la memoria para cada ejecucin. Un buen punto de partida podra ser el valor predeterminado que se muestra en Opciones de ejecucin, y a partir de ah, ajustar hacia arriba o hacia abajo para encontrar el punto ptimo en trminos de tiempo, para completar todas las ejecuciones.

Sexta: cada ejecucin en paralelo tiene su propio mundo en el que se ejecuta el modelo. Este mundo no se borra automticamente por Espacio-Conductual si una ejecucin en paralelo se reutiliza para otra repeticin, lo que ocurre con bastante frecuencia. En otras palabras, es importante tener cuidado con los datos que no se borran.

Sptima: hay una posibilidad extremadamente pequea de que, al iniciar mltiples ejecuciones en paralelo al mismo tiempo, stas puedan terminar con el mismo estado del generador de nmeros aleatorios. Esto significara que las ejecuciones produciran resultados idnticos en todas las operaciones aleatorias y, probablemente, resultados iguales en general. Esto podra ocurrir al ejecutar en procesadores muy rpidos y con muchas ejecuciones en paralelo al mismo tiempo. Si se desea garantizar que esto no afecte los resultados, puede agregarse el comando random-seed new-seed a los comandos de configuracin para generar una nueva semilla aleatoria nica en cada ejecucin. De hecho, guardar new-seed como una variable global permitira volver a ejecutar una ejecucin en otro momento, utilizando ese valor para establecer random-seed manualmente.

Es posible ejecutar experimentos de Espacio-Conductual desde la lnea de comandos, sin ninguna interfaz grfica de usuario (headless). Esto resulta til para automatizar ejecuciones en una sola mquina o en un conjunto de mquinas.

Cmo se usa

Se ejecuta NetLogo usando la aplicacin NetLogo_Console con el argumento de lnea de comandos --headless sta se encuentra en el directorio principal de la instalacin de NetLogo. El script NetLogo_Console es compatible con los siguientes argumentos:

  • --headless: activa el modo sin interfaz para ejecutar un experimento de Espacio-Conductual (obligatorio, de lo contrario se abrir la interfaz grfica).
  • --model <ruta>: ruta de archivo del modelo a abrir (obligatorio).
  • --setup-file <ruta>: lee las configuraciones de experimentos desde este archivo en lugar del archivo del modelo.
  • --experiment <nombre>: nombre del experimento a ejecutarse.
  • --table <ruta>: ruta para enviar resultados tabulados (o - para resultados estndar).
  • --spreadsheet <ruta>: ruta para enviar resultados en hoja de clculo (o - para resultados estndar)
  • --lists <ruta>: ruta para enviar resultados en lista (o - para resultados estndar), no puede utilizarse sin --table o --spreadsheet.
  • --stats <ruta>: ruta para enviar resultados estadsticos (o - para resultados estndar), no puede utilizarse sin --table o --spreadsheet
  • --threads <nmero>: utiliza esta cantidad de hilos para realizar ejecuciones de modelo en paralelo, o 1 para desactivar ejecuciones en paralelo. El valor predeterminadao es floor(0.75 * <number of processors>).
  • --update-plots: habilita las actualizaciones de grficos. Se incluye si se desea exportar datos de grficos o se excluye para mejorar el rendimiento.
  • --min-pxcor <nmero>: reemplaza la configuracin de tamao del mundo en el archivo del modelo.
  • --max-pxcor <nmero>: reemplaza la configuracin de tamao del mundo en el archivo del modelo.
  • --min-pycor <nmero>: reemplaza la configuracin de tamao del mundo en el archivo del modelo.
  • --max-pycor <nmero>: reemplaza la configuracin de tamao del mundo en el archivo del modelo.

--model es obligatorio. Si no se especifica --experiment, se debe especificar --setup-file. Por defecto, no se generan resultados, por lo que generalmente se deber especificar --table o --spreadsheet, o ambos. Si se especifica alguna de las dimensiones del mundo, se deben proporcionar las cuatro.

Es importante tener en cuenta que antes de NetLogo 6.3.0, las indicaciones recomendaban el uso de netlogo-headless.sh (o netlogo-headless.bat en Windows) junto con una instalacin independiente de Java en el sistema para llevar a cabo experimentos de Espacio-Conductual. Sin embargo, en una computadora personal, el mtodo recomendado es utilizar la aplicacin NetLogo_Console --headless. Dado que NetLogo_Console emplea la versin de Java que se incluye con NetLogo, no es necesario realizar una instalacin adicional de software ni configuraciones especiales.

Configuracin de experimentos en XML

An no contamos con documentacin detallada sobre cmo crear configuraciones de experimentos en XML. Sin embargo, si se tiene familiaridad con XML, los siguientes consejos pueden ayudar a comenzar.

La estructura de las configuraciones de experimentos de Espacio-Conductual en XML se define mediante un archivo de Definicin de Tipo de Documento (DTD). <experiments>
  <experiment name="My WSP Exploration" repetitions="5" runMetricsEveryStep="false">

    <go>go</go>
    <timeLimit steps="2000"/>
    <metric>count wolves</metric>
    <metric>count sheep</metric>
    <metric>sum [grass-amount] of patches</metric>
    <enumeratedValueSet variable="energy-gain-from-grass">
      <value value="2"/>
    </enumeratedValueSet>
    <steppedValueSet variable="number-of-wolves" first="5" step="1" last="15"/>
    <enumeratedValueSet variable="movement-cost">
      <value value="0.5"/>
    </enumeratedValueSet>
    <enumeratedValueSet variable="energy-gain-from-sheep">
      <value value="5"/>
    </enumeratedValueSet>
    <enumeratedValueSet variable="number-of-sheep">
      <value value="500"/>
    </enumeratedValueSet>
    <enumeratedValueSet variable="grass-regrowth-rate">
      <value value="0.3"/>
    </enumeratedValueSet>

En este ejemplo, se muestra solo una configuracin de experimento, pero pueden incluirse tantas como se deseen entre las etiquetas de inicio y finalizacin experiments

Al observar la DTD y los ejemplos creados en la interfaz grfica, esperamos que sea evidente cmo utilizar las etiquetas para especificar diferentes tipos de experimentos. La DTD especifica cules etiquetas son obligatorias y cules son opcionales, cules pueden repetirse y cules no, entre otros detalles.

Si se desea crear un archivo de configuracin para NetLogo 6.3.0 y versiones anteriores, para las cuales Exportar no est disponible, debe tenerse en cuenta que en un archivo de modelo, el XML para configuraciones de experimentos no comienza con encabezados XML, ya que no todo el archivo es XML, solo una parte. Adems, se debe iniciar el archivo con los encabezados XML adecuados, como se muestra a continuacin:

La segunda lnea debe incluirse tal como se muestra. En la primera lnea, puede especificarse una codificacin diferente a UTF-8, como ISO-8859-1.

NetLogo asigna un mximo de la mitad de la memoria total del sistema para ejecutar el modelo, segn sea necesario. Si desea establecerse una cantidad mxima de memoria para el uso de Espacio-Conductual, puede utilizarse la configuracin -Xmx para especificar el tamao del heap

Observa el uso de -Dfile.encoding=UTF-8. Esto obliga a todos los archivos de E/S a utilizar la codificacin UTF-8. Al hacerlo, se garantiza que NetLogo pueda cargar todos los modelos de manera coherente y que las primitivas de archivo file-* funcionen de manera uniforme en todas las plataformas, incluyendo modelos que contengan caracteres Unicode.

Control API

Si Espacio-Conductual no es suficiente para tus necesidades, una posible alternativa es utilizar nuestra API de Control, que permite escribir cdigo en Java para controlar NetLogo. La API permite ejecutar experimentos de Espacio-Conductual desde el cdigo en Java, o bien escribir cdigo personalizado que controle NetLogo de manera ms directa para realizar tareas similares a las de Espacio-Conductual. Consulta la seccin Control del Manual del Usuario para obtener ms detalles sobre ambas posibilidades.
